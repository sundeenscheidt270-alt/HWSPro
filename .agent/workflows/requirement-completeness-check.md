# 需求调研完整性检查清单

**使用场景**: 调研后检查是否有遗漏，确保需求调研的完整性

---

## 检查前准备
- [ ] 准备好调研报告
- [ ] 准备好访谈记录（如有）
- [ ] 准备好录音/截图（如有）

---

## 第一部分：业务理解完整性

### 1.1 业务背景
- [ ] 是否清楚业务模式？
- [ ] 是否了解涉及的部门和角色？
- [ ] 是否获取了业务量级数据？
- [ ] 是否理解为什么要做这个需求？

### 1.2 业务目标
- [ ] 目标是否可量化？
- [ ] 是否有明确的成功指标？
- [ ] 是否有优先级排序？

---

## 第二部分：流程场景完整性

### 2.1 主流程
- [ ] 是否梳理了完整的端到端流程？
- [ ] 流程的起点和终点是否明确？
- [ ] 各环节的操作人是否明确？
- [ ] 各环节的耗时是否了解？

### 2.2 异常流程
- [ ] 是否考虑了失败场景？（如网络超时）
- [ ] 是否考虑了取消/撤销场景？
- [ ] 是否考虑了驳回场景？
- [ ] 是否考虑了数据冲突场景？

### 2.3 业务场景覆盖
- [ ] 是否覆盖了高频场景？（日常）
- [ ] 是否覆盖了特殊时期场景？（月末/年末/促销）
- [ ] 是否覆盖了批量操作场景？
- [ ] 是否覆盖了边界场景？（第一单/最后一单）

---

## 第三部分：数据规则完整性

### 3.1 数据对象
- [ ] 核心业务对象是否识别完整？
- [ ] 对象之间的关系是否清楚？（1:1/1:N/N:N）
- [ ] 是否明确了数据来源？（用户输入/系统生成/其他系统）

### 3.2 字段信息
- [ ] 必填字段是否明确？
- [ ] 字段类型是否明确？（文本/数字/日期/枚举）
- [ ] 字段长度限制是否明确？
- [ ] 字段校验规则是否明确？
- [ ] 枚举值是否穷举？

### 3.3 业务规则
- [ ] 是否梳理了关键业务规则？
- [ ] 规则的触发条件是否明确？
- [ ] 规则是否有例外情况？
- [ ] 规则冲突时如何处理？
- [ ] 规则是固定的还是可配置的？

### 3.4 计算逻辑
- [ ] 金额计算逻辑是否明确？
- [ ] 税费计算逻辑是否明确？
- [ ] 折扣计算逻辑是否明确？
- [ ] 库存扣减逻辑是否明确？
- [ ] 计算精度是否明确？（小数点几位）

---

## 第四部分：交互体验完整性

### 4.1 查询与筛选
- [ ] 查询条件有哪些？
- [ ] 默认值是什么？
- [ ] 是否支持模糊搜索？
- [ ] 是否支持高级筛选？

### 4.2 列表展示
- [ ] 列表需要展示哪些字段？
- [ ] 默认排序规则是什么？
- [ ] 是否需要分页？每页多少条？
- [ ] 是否需要导出？

### 4.3 表单录入
- [ ] 必填项是否明确？
- [ ] 是否有默认值？
- [ ] 是否有字段联动？
- [ ] 是否有自动计算？
- [ ] 错误提示文案是否明确？

### 4.4 操作按钮
- [ ] 有哪些操作按钮？
- [ ] 各按钮的权限要求是否明确？
- [ ] 各按钮在什么状态下可用？
- [ ] 是否需要二次确认？（如删除）

---

## 第五部分：权限协同完整性

### 5.1 角色权限
- [ ] 涉及哪些角色？
- [ ] 各角色的职责是否明确？
- [ ] 各角色的操作权限是否明确？
- [ ] 数据范围权限是否明确？（只看自己的/部门的/全部）
- [ ] 字段级权限是否明确？（可见/可编辑/脱敏）

### 5.2 审批流程
- [ ] 是否需要审批？
- [ ] 审批规则是什么？（按金额/按类型）
- [ ] 几级审批？
- [ ] 审批人如何确定？
- [ ] 审批超时如何处理？
- [ ] 是否支持加签/转审？

### 5.3 通知协同
- [ ] 需要通知谁？
- [ ] 什么时候通知？
- [ ] 通知方式？（站内信/短信/邮件/企微）
- [ ] 通知内容模板？

---

## 第六部分：集成边界完整性

### 6.1 系统集成
- [ ] 需要对接哪些系统？
- [ ] 数据从哪个系统获取？
- [ ] 数据推送到哪个系统？
- [ ] 主数据归属是否明确？
- [ ] 是同步调用还是异步调用？

### 6.2 数据一致性
- [ ] 多系统数据如何保持一致？
- [ ] 是强一致还是最终一致？
- [ ] 数据冲突如何处理？
- [ ] 是否需要对账机制？

---

## 第七部分：非功能需求完整性

### 7.1 性能要求
- [ ] 响应时间要求？（如<3秒）
- [ ] 并发量要求？（如支持100人同时操作）
- [ ] 批量处理能力？（如一次导入1000条）

### 7.2 数据安全
- [ ] 哪些数据需要脱敏？
- [ ] 哪些操作需要审计留痕？
- [ ] 数据保留多久？

### 7.3 兼容性
- [ ] 是否需要支持移动端？
- [ ] 浏览器兼容要求？
- [ ] 是否需要支持离线？

---

## 第八部分：历史数据完整性

### 8.1 数据迁移
- [ ] 是否需要迁移历史数据？
- [ ] 迁移范围？（如近3年数据）
- [ ] 数据清洗规则？
- [ ] 如何验证迁移正确性？

### 8.2 数据归档
- [ ] 历史数据如何归档？
- [ ] 归档后还能查询吗？
- [ ] 保留期限？

---

## 检查完成后的行动

### 如果发现遗漏（重要）
1. 记录遗漏项
2. 评估影响（是否影响方案设计）
3. 决定是否需要补充调研

**补充调研Prompt**：
```
@requirement-research-prompt.md

我在需求调研中发现以下遗漏：
1. [遗漏项1]
2. [遗漏项2]

请为这些遗漏项生成补充调研问题
```

### 如果通过检查
- 更新调研报告状态为"已完成"
- 进入需求文档编写阶段
- 使用 @prd-template.md 开始写PRD

---

## 常见遗漏提醒

⚠️ **经常被遗漏的8个点**：
1. 异常流程（只关注了正常流程）
2. 边界场景（只关注了主流场景）
3. 计算精度（金额小数点保留几位）
4. 字段联动（A字段变化时B字段如何变）
5. 审批超时（审批人一直不审怎么办）
6. 数据冲突（两人同时修改）
7. 历史数据迁移（新系统上线后老数据怎么办）
8. 移动端需求（是否需要手机上操作）
