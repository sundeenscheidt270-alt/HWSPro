# 需求上线自查清单

**使用场景**: 需求开发完成后，上线前、上线中、上线后的全流程自查，确保上线质量和用户体验。

---

## 阶段1：上线前 - 功能确认

### 1.1 需求范围确认
- [ ] 本次上线的需求清单是否明确？
- [ ] 需求是否与PRD一致？有无变更？
- [ ] 变更内容是否已同步给所有相关方？
- [ ] 是否有需求被砍掉或延期？原因是否记录？

### 1.2 测试完成确认
- [ ] 所有功能是否都通过测试？
- [ ] 测试用例覆盖率是否达标？
- [ ] 是否有遗留Bug？严重程度如何？
- [ ] 遗留Bug是否有明确的处理计划？
- [ ] UAT（用户验收测试）是否通过？

### 1.3 关联影响确认
- [ ] 是否影响其他模块？影响范围是否评估？
- [ ] 是否影响其他系统？是否已协调？
- [ ] 是否影响历史数据？兼容性是否验证？
- [ ] 是否影响现有用户操作习惯？

---

## 阶段2：上线前 - 技术准备

### 2.1 数据库变更
- [ ] 是否有数据库结构变更（DDL）？
- [ ] DDL脚本是否已评审？
- [ ] 是否需要数据迁移？迁移脚本是否准备好？
- [ ] 数据迁移是否在测试环境验证通过？
- [ ] 迁移预计耗时？是否需要停服？

### 2.2 配置变更
- [ ] 是否有配置项变更？
- [ ] 新增配置项的默认值是否合理？
- [ ] 配置变更是否记录在案？

### 2.3 依赖确认
- [ ] 是否依赖其他系统先上线？
- [ ] 依赖的接口是否已就绪？
- [ ] 第三方服务是否正常？

### 2.4 回滚方案
- [ ] 是否准备了回滚方案？
- [ ] 回滚脚本是否准备好？
- [ ] 回滚操作步骤是否明确？
- [ ] 回滚后数据如何处理？

---

## 阶段3：上线前 - 发布计划

### 3.1 发布时间
- [ ] 发布时间是否确定？
- [ ] 是否避开业务高峰期？
- [ ] 是否需要停服？停服时长？
- [ ] 停服公告是否准备好？

### 3.2 发布方式
- [ ] 是全量发布还是灰度发布？
- [ ] 灰度范围是什么？（按租户/按用户/按比例）
- [ ] 灰度观察期多久？
- [ ] 全量发布的条件是什么？

### 3.3 人员安排
- [ ] 发布负责人是谁？
- [ ] 开发值班人员是谁？
- [ ] 测试值班人员是谁？
- [ ] 产品值班人员是谁？
- [ ] 紧急联系方式是否同步？

---

## 阶段4：上线执行

### 4.1 发布前检查
- [ ] 代码是否已合并到发布分支？
- [ ] 发布包是否已构建成功？
- [ ] 发布环境是否就绪？

### 4.2 发布执行
- [ ] 数据库脚本是否按顺序执行？
- [ ] 应用是否部署成功？
- [ ] 配置是否生效？
- [ ] 服务是否正常启动？

### 4.3 发布后快速验证
- [ ] 核心功能是否可用？
- [ ] 页面是否正常加载？
- [ ] 接口是否正常响应？
- [ ] 日志是否有异常？
- [ ] 监控告警是否正常？

---

## 阶段5：上线后 - 验收确认

### 5.1 功能验收
- [ ] 所有上线功能是否逐一验证？
- [ ] 新功能是否符合PRD预期？
- [ ] 老功能是否正常（回归验证）？
- [ ] 边界场景是否验证？

### 5.2 数据验收
- [ ] 数据迁移是否正确？
- [ ] 新老数据是否兼容？
- [ ] 报表数据是否正常？
- [ ] 是否需要数据核对？

### 5.3 性能验收
- [ ] 页面加载速度是否正常？
- [ ] 接口响应时间是否达标？
- [ ] 是否有慢查询？
- [ ] 服务器资源是否正常？

### 5.4 监控确认
- [ ] 错误日志是否有异常增长？
- [ ] 告警是否正常触发？
- [ ] 业务指标是否正常？

---

## 阶段6：上线后 - 用户通知

### 6.1 更新公告
- [ ] 更新公告是否发布？
- [ ] 公告内容是否清晰易懂？
- [ ] 公告渠道是否覆盖所有用户？（站内信/邮件/企微/钉钉）
- [ ] 重要变更是否重点标注？

### 6.2 帮助文档
- [ ] 新功能帮助文档是否更新？
- [ ] 操作指南是否配图？
- [ ] FAQ是否更新？
- [ ] 视频教程是否需要？

### 6.3 培训安排
- [ ] 是否需要用户培训？
- [ ] 培训时间是否安排？
- [ ] 培训材料是否准备好？
- [ ] 是否需要录制培训视频？

---

## 阶段7：上线后 - 观察与收尾

### 7.1 观察期
- [ ] 观察期时长是多久？（建议至少3天）
- [ ] 观察期内是否有专人值班？
- [ ] 用户反馈渠道是否畅通？
- [ ] 问题响应机制是否明确？

### 7.2 问题处理
- [ ] 上线后问题是否及时记录？
- [ ] 问题是否分类（Bug/体验/建议）？
- [ ] 紧急问题是否有快速修复流程？
- [ ] 是否需要发布补丁版本？

### 7.3 收尾工作
- [ ] 上线总结是否完成？
- [ ] 经验教训是否记录？
- [ ] 相关文档是否归档？
- [ ] JIRA/禅道等任务是否关闭？
- [ ] 是否通知相关方上线完成？

---

## 检查完成后的行动

### 如果发现阻塞项
1. 立即评估影响
2. 决定是否延期上线
3. 同步给所有相关方
4. 记录延期原因和新计划

### 如果发现风险项
1. 评估风险等级（高/中/低）
2. 制定风险应对措施
3. 明确风险责任人
4. 决定是否继续上线

### 如果全部通过
1. 确认发布时间
2. 通知所有相关方
3. 进入发布执行阶段

---

## 常见踩坑提醒

⚠️ **上线常见问题Top 10**：

1. **数据迁移遗漏** - 只迁移了结构，忘了迁移数据
2. **配置未生效** - 改了配置但没重启服务
3. **缓存未清理** - 老数据还在缓存里
4. **权限未配置** - 新功能没给用户开权限
5. **依赖未就绪** - 依赖的系统还没上线
6. **回滚方案缺失** - 出问题了不知道怎么回滚
7. **通知不到位** - 用户不知道有新功能
8. **文档未更新** - 帮助文档还是老版本
9. **监控未配置** - 出问题了没有告警
10. **灰度范围不当** - 灰度用户选错了，没测到关键场景
